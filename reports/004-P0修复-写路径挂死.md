## P0-1 · 写路径挂死（watch_intent 经 MCP 4 分钟无响应）——✅ 已修复

### 结论（根因）
**不是 `watch_intent` 单函数 bug，是 SQLite 写锁 + 长驻 stdio server 的读写争用。**

真机数据库处于 SQLite 默认的 **rollback-journal（`journal_mode=delete`）** 模式。该模式下 **一个未结束的读事务会阻塞任何写事务**。长驻的 `ohp serve` 进程会把「流式读」（`search_jobs`/`check_watches` 遍历 11,825 行，持有 SHARED 读锁）与「写」（`watch_intent`/`apply` 的 INSERT）交错执行；当写落在一个仍活跃的读之后，写就卡在 busy timeout 上，在 async 事件循环里表现为「几分钟无响应」。

`apply` 当时秒回，是因为那一刻**没有重叠的读事务**，写抢锁成功——这与「锁争用 bug」完全自洽，也正好证伪了「写路径整体挂死」。探针帮了大忙。

### 证据（可复现）
用真库副本做对照实验：

| 场景 | 结果 |
|---|---|
| rollback-journal 下，读事务活跃时发起写 | **5.46s 后失败 `OperationalError: database is locked`** |
| WAL 下，读事务活跃时发起写 | **0.004s 提交成功** |

单进程无争用时 `watch_intent` 本就 0.02s 返回（所以 CLI `ohp watch`、既有测试都没暴露它）——bug 只在「真库体量 + 并发读写」下现形。

### 修复
`src/openhire/db/session.py` 的 SQLite 引擎改为：
- `PRAGMA journal_mode=WAL` —— 一个写可与多个读并发，读不再阻塞写。
- `PRAGMA busy_timeout=5000` —— 真有写-写争用时有界等待，而非瞬间报错或无限挂。
- `PRAGMA synchronous=NORMAL` —— WAL 下的安全且高效档位。
- `connect_args={"check_same_thread": False, "timeout": 30}` —— 允许连接池连接跨 FastMCP 工作线程复用。

WAL 是**持久**设置（写入 DB 头），真库已转为 WAL（生成 `-wal`/`-shm` 边车文件，正常现象）。

### 验证
1. 新增 `tests/test_concurrency.py`（2 项）：
   - 引擎确为 WAL 且 busy_timeout>0；
   - **持有一个真实读事务时，另一线程的 `watch_intent` 写必须 < 2s 完成**（复现 P0-1 症结，防回归）。
2. 全量 `pytest`：**100 passed**。
3. **T3 真库重跑**（fingerprint `88ba1102edb9205d`，4 线程持续 `search_jobs` 读压之下，走真实 MCP 协议）：

   0.046s 返回，只落匿名指纹 + 过滤条件，隐私红线①未破。

## P0-2 · datePosted 造假 + ghost_score 恒 0 ——✅ 已修复

### 根因（真库确诊）
- 入库时 `_insert_new_job` 把 `first_seen_at`/`verified_at` 都写成**抓取时刻**，而 ATS 解析出的真实发布日（`JobRecord.posted_at`）因 `Job` 表**没有对应列**被直接丢弃。
- 全库 11,825 条 `first_seen_at` **只有 1 个 distinct 值** = `2026-07-10 01:51:46.387274` → `datePosted` 对所有岗位都是这一天（造假坐实）。
- ghost_score 用 `first_seen_at`（岗龄≈0 天）算，且 `relist_count` 全 0 → **全库 ghost_score 恒 0**。
- **DeepSeek 未污染日期**：`extraction_source='deepseek'` 但日期就是那次唯一抓取的 `now`；重建只动了 skills/remote/salary。你观察到的「4 家 last_crawled_at 微秒级相同」是 `last_crawled_at` 全库仅 2 个 distinct 值（批次 `now` 被同一次运行的所有公司共享）——属 API 定型 #5，非重建 bug。

### 修复
1. `Job` 新增 `posted_at` / `updated_at` 列（真实 ATS 发布/更新日）；`db/migrate.py` 幂等补列。
2. 三家 ATS 解析器本就产出 `posted_at`（Greenhouse `first_published`、Lever `createdAt`、Ashby `publishedAt`）——`ingest.py` 现在把它们落库，并让 ghost_score 以真实发布日为岗龄锚点（无则回退 `first_seen_at`）。
3. `service.job_posting`：`datePosted` 改用真实 `posted_at`（回退 `first_seen_at`），新增 **`days_open`**（岗龄天数）。
4. 新增 `ohp backfill-dates`（`pipeline/backfill.py`）：**免费**重抓公开 ATS，为存量岗位回填真实发布日并重算 ghost_score，**绝不重跑抽取、不动 skills/salary/remote**。
