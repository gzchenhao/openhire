## 1) 首跑数据方案（你已拍板：混合）

- **快照 = GitHub Release 资产，不进 wheel。** 只含 `companies` + `jobs` 两表；`watches`/`applications` 等任何用户态**零行**（实际上快照里根本不创建这两张表）。
- **`ohp bootstrap` 默认：** 下载快照 → 就地增量抓取一次刷新 `verified_at`/下线 → 打印快照龄与刷新结果。
- **`--fresh`：** 跳过快照，现抓公开 ATS 建库（启发式、免费）。
- **`--deepseek`：** 自带 key 的高质量抽取。
- **开跑前先声明将做什么**（拉 ~20MB 公开职位数据 + 抓 96 家公开 ATS · 无账号 · 无 PII），确认后才动。

**为什么不用「随包带快照」(方案 A)：** 数据一出厂即过时（days_open/ghost_score 漂移、已下线岗仍显示在架），与协议「新鲜度/真实性」核心承诺冲突；且把 80MB+ 数据塞进代码 wheel 是反模式。混合方案兼得「秒开」与「新鲜」。

### 实现
- `pipeline/snapshot.py`：`build_snapshot`（按源库 DDL 只复制两表，**构建时强制零用户态，违规即 raise 失败**）+ `install_snapshot`（下载/解压 + 安装前再次校验零用户态，绝不写入用户态）。
- `cli.py`：`ohp bootstrap`（含 `--fresh/--deepseek/--force/--yes/--snapshot-url`）+ `ohp snapshot-build`（维护者构建）。
- `db/session.py`：`dispose_engine()`（安装快照前释放 SQLite 文件句柄，解决 Windows 覆盖锁）。
- `config.py`：`OPENHIRE_SNAPSHOT_URL`（默认 GitHub Release URL，可覆盖）。

### 证据（真机实跑）
- **构建真快照**（源库含 4 条 watch / 5 条 application）：`公司 96 · 职位 11825 · 13.2MB gz`，**快照内 watches/applications 表数=0**，零用户态校验通过 → 证明用户态被彻底剥离。
- **bootstrap 全流程**（新空库 + 本地快照 URL）：预声明 → 安装（96/11825，龄 3 天）→ 增量刷新（新增 323 · 更新 340 · 下线 452 · 96 家）→ ✓。
- **护栏**：已有索引再跑 bootstrap 提示「加 --force」；`--fresh` 预声明 + 取消可用。
- **红线测试** `tests/test_snapshot.py`（4 项）：快照剥离全部用户态 / 空索引拒绝构建 / 安装拒收含用户态的快照 / 构建→安装往返（role_family 等字段完好）。
